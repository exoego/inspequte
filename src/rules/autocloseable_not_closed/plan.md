# AUTOCLOSEABLE_NOT_CLOSED — Plan

## Problem Framing

Detect locally created objects implementing `java.lang.AutoCloseable` that can reach a method exit without `close()`
being called on every reachable path.

Typical problematic pattern:

```java
void leak() {
    InputStream in = new FileInputStream("f.txt");
    // exits without close() → resource leak
}
```

Java's try-with-resources and Kotlin's `.use {}` are the standard solutions.
At the bytecode level both compile to `close()` calls inside `finally` blocks, so detection logic can handle them
uniformly.

## Detection Strategy

Follows the same structure as the `executor_service_not_shutdown` rule.

### Phase 1: Type Identification

- Determine whether the type of an object created by a `NEW` instruction, or returned by an `INVOKESTATIC` /
  `INVOKEVIRTUAL` call, implements `java.lang.AutoCloseable` or `java.io.Closeable` via BFS traversal of the type
  hierarchy.
- For external classes absent from the class map, fall back to a hardcoded list of well-known JDK AutoCloseable
  implementations (`InputStream`, `OutputStream`, `Reader`, `Writer`, `Connection`, `Statement`, `ResultSet`, `Channel`
  families, etc.).
- Exclude types whose `close()` method is virtually no-op, from a hardcoded list: `java.io.CharArrayWriter`,
  `java.io.CharArrayReader`, `java.io.StringWriter`, `java.io.StringReader`, `java.io.ByteArrayOutputStream`,
  `java.io.ByteArrayInputStream`, `java.io.StringBufferInputStream`
- Exclude types whose `close()` method is no-op for most cases, to reduce lots of false positives, from a hardcoded
  list: `java.util.stream.Stream`, `java.util.stream.IntStream`, `java.util.stream.LongStream`,
  `java.util.stream.DoubleStream`.

### Phase 2: Symbolic Tracking (Intraprocedural)

- Worklist-based dataflow analysis using `WorklistSemantics` + `StackMachine`.
- When an AutoCloseable is stored into a local variable after `NEW` + `<init>` or a factory method call, generate a
  symbol and add it to the active set.
- When `close()` is called on a receiver matching a tracked symbol, remove that symbol from the active set.

### Phase 3: Escape Analysis

Remove a symbol from the active set (ownership transferred out of the method) when:

- Stored into a field (`PUTFIELD` / `PUTSTATIC`)
- Stored into an array (`AASTORE`)
- Passed as an argument to another method (`INVOKE*` argument position)
- Returned via `ARETURN`
- Passed as a constructor argument to another AutoCloseable (wrapper pattern:
  `new BufferedReader(new FileReader(...))` — inner resource delegates to outer)

### Phase 4: Exit Path Inspection

- At normal exits (`RETURN` / `ARETURN` / `IRETURN` / etc.) and exceptional exits (`ATHROW`), report a finding for each
  symbol still in the active set.
- The reporting condition is "at least one reachable exit path from the creation site lacks a `close()` call" (i.e.,
  `close()` must be guaranteed on all paths).

## Scope

### In scope

- AutoCloseable instances locally created within a method
- Direct creation via `NEW` + constructor
- Creation factory methods returning AutoCloseable
- Detection of `close()` calls as the release mechanism
- Bytecode-level `close()` generated by try-with-resources / Kotlin `.use {}`
- Wrapper pattern (inner AutoCloseable delegates to outer)
- Safe exclusion via escape / ownership transfer

### Non-goals

- Interprocedural analysis (tracking `close()` in helper methods)
- Tracking AutoCloseables received from parameters, fields, or method return values
- `@Suppress` / `@SuppressWarnings` suppression
- Semantic changes based on non-JSpecify annotations
- Custom close methods (`release()`, `dispose()`, etc.)

## Test Strategy

### TP (should report)

- Method returns without calling `close()`
- Method calls `close()` on only one branch of a conditional (early return pattern)
- Method leaks `close()` on an exception path
- Method creates multiple AutoCloseables but only closes some

### TN (should not report)

- Resource managed by try-with-resources
- Resource managed by Kotlin `.use {}`
- Resource closed in a `finally` block
- Resource stored into a field (escape)
- Resource passed as an argument to another method (escape)
- Resource returned from the method (escape)
- Wrapper pattern (`new BufferedReader(new FileReader(...))`)

### Edge

- `close()` called only inside a null-check conditional
- AutoCloseable created inside a loop
- Same local variable slot reused for different resources

## Complexity

- Time: O(B × S) — B = number of basic blocks in the method, S = number of active symbols
- Comparable cost to `executor_service_not_shutdown`
- Worklist convergence is guaranteed (symbol removal is monotonic)

## Determinism

- Use `BTreeMap` / `BTreeSet` for symbols and findings to guarantee iteration order
- Output findings sorted by (class name, method name, method descriptor, bytecode offset)
  to ensure identical output for identical input

## Risks

- [ ] **Missing type information**: Types from external dependencies absent in the class map cannot be resolved via
  hierarchy traversal. Relies on hardcoded list coverage.
  → Prioritize precision; accept false negatives in the initial release.
- [ ] **Chained wrappers**: In `new A(new B(new C(...)))`, intermediate resource B may not be recognized as escaped.
  → Mitigate by checking AutoCloseable types in constructor arguments.
- [ ] **CFG precision for exception handlers**: Implicit exception edges may increase false positives.
  → Follow the same CFG strategy as the executor rule; tune during verification.
- [ ] **Performance on large methods**: Symbol count may grow in methods with thousands of instructions.
  → Bound via existing `StackMachine` budget constraints.
