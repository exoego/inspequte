# UNUSED_LAMBDA_PARAMETERS — Plan

## Problem Framing

Java and Kotlin lambda expressions (and SAM-converted anonymous functions) sometimes declare parameters that are never
read in the lambda body. Unused parameters increase cognitive load, obscure the lambda's actual data dependencies, and
can mask copy-paste errors where the wrong parameter is referenced.

Unused parameters can also be a sign of bugs, where the outside variables are passed wrongly:

```kotlin
folder.children.forEach {
    // an implicit single parameter "it" (or "child" if named explicitly) should be used,
    // but instead, the parent folder is passed wrongly and type check passes
    DoSomethingWithFolder(folder)
}
```

In Kotlin, the idiomatic fix is to replace the unused parameter with `_`. In Java (22+, JEP 456), unnamed variables
serve the same purpose. For earlier Java versions the parameter cannot be omitted, but the finding still highlights
unnecessary coupling to an API surface.

## Detection Strategy

### Bytecode-level approach

1. **Identify lambda methods** — Scan for methods that are lambda implementation targets:
    - Java: private synthetic methods named `lambda$<enclosingMethod>$<n>`, referenced by `invokedynamic` bootstrap via
      `LambdaMetafactory`.
    - Kotlin: `invoke` / `invokeSuspend` methods inside anonymous inner classes that implement
      `kotlin.jvm.functions.FunctionN` or SAM interfaces generated by the Kotlin compiler.

2. **Determine parameter slots** — For each identified lambda method, enumerate its declared parameters, excluding
   captured variables / closure context:
    - **Java**: Captured variables appear as **leading parameters** in the synthetic lambda method. Distinguish them by
      cross-referencing the `invokedynamic` call-site descriptor (captured args) vs. the functional interface method
      descriptor (lambda params). Only the trailing parameters matching the functional interface are lambda params.
    - **Kotlin (non-inline)**: Captured variables are stored as **fields** of the anonymous inner class, not as
      parameters of `invoke`. All parameters of `invoke` / `invokeSuspend` are lambda params (except the synthetic
      `$result: Object` parameter in `invokeSuspend` for suspend lambdas — this must be excluded).

3. **Scan for LOAD instructions** — Walk the method's instruction list and record which parameter slots are loaded (
   `ALOAD`, `ILOAD`, `LLOAD`, `FLOAD`, `DLOAD`). A parameter whose slot is never loaded is unused.

4. **Emit one finding per unused parameter** — Report the parameter index, the enclosing lambda method, and the
   enclosing class.

### Kotlin inline lambda approach

Kotlin `inline` functions (e.g. `forEach`, `map`, `let`) inline the lambda body directly into the caller method.
No separate class or method is generated. However, the Kotlin compiler inserts debug markers in the
`LocalVariableTable` that delineate inline boundaries:

- `$i$f$<functionName>` — marks the inlined function body range (start/end labels + slot).
- `$i$a$-<functionName>-<caller>$<lambda>$<index>` — marks the inlined **lambda body** range.
- The lambda parameter itself appears as a named local variable (e.g. `it`, or the user-chosen name) with a
  scope that overlaps the `$i$a$` range.

Detection steps for inline lambdas:

1. **Find `$i$a$` markers** in the `LocalVariableTable` of each method.
2. **Identify lambda parameter variables** — local variables whose scope overlaps the `$i$a$` range, excluding
   the marker variables themselves, `this`, and captured outer variables.
3. **Scan LOAD instructions within the `$i$a$` range** — if the lambda parameter slot is never loaded within
   that instruction range, the parameter is unused.
4. **Emit finding** — attributed to the enclosing method and the inline call site line.

This approach depends on debug info (`-g` / default Kotlin compiler settings). If `LocalVariableTable` is stripped,
inline lambda analysis is silently skipped (no false positives).

### `_` convention (Kotlin and Java 22+)

- If the parameter name is `_` (or Kotlin's `$noName_<n>`), treat it as intentionally unused and do **not** report it.
  This applies to both Kotlin's `_` convention and Java 22+ unnamed variables (JEP 456). This avoids double-flagging
  what the developer has already acknowledged.

## Scope

- Java lambda expressions compiled via `LambdaMetafactory`.
- Kotlin lambda expressions (non-inline) compiled to anonymous inner classes.
- Kotlin implicit `it` parameter — when a single-parameter lambda never references `it`, the parameter slot is still
  present in the `invoke` method and detected by the same LOAD scan.
- Kotlin inline lambdas (e.g. `forEach`, `map`, `let`) — detected via `LocalVariableTable` debug markers
  (`$i$a$` / `$i$f$`) that the Kotlin compiler emits. Silently skipped if debug info is stripped.
- SAM conversions in both languages.

## Non-Goals

- **Anonymous inner classes written by hand** (non-SAM, non-lambda) — These are regular classes; unused parameter
  detection in them is a different rule.
- **Method references** — Method references (`Class::method`) do not introduce a new parameter scope; no parameters can
  be "unused" in a method reference.
- **`@Suppress`-based suppression** — Not supported per project annotation policy.
- **Non-JSpecify annotation semantics** — Per project annotation policy, only JSpecify annotations are recognized.
  Non-JSpecify annotations do not change rule behavior.
- **Multi-expression lambda body data-flow** — No inter-procedural or complex data-flow analysis. Only local LOAD
  instruction presence is checked.

## Complexity

- **Non-inline path**: O(M * P) where M = number of lambda methods, P = average instruction count per method.
- **Inline path**: O(N * L) where N = total methods in all classes, L = average `LocalVariableTable` size. The scan
  for `$i$a$` markers requires visiting every method's local variable table, but the per-method cost is small
  (table size is typically proportional to method length).
- Both paths are effectively linear in total bytecode size.
- **Space**: O(S) where S = number of parameter slots across all lambda methods (small constant per method).
- No whole-program data-flow or CFG construction required.

## Determinism

- **Non-inline lambda findings** are keyed by (class name, method name, method descriptor, parameter index) — all
  stable across runs.
- **Inline lambda findings** are keyed by (enclosing class name, enclosing method name, enclosing method descriptor,
  `$i$a$` marker name, parameter name) — the marker name encodes the inline call identity and is stable for a given
  source structure.
- Iteration order follows class file discovery order (deterministic in inspequte).

## Risks

- [ ] **Captured-vs-lambda parameter disambiguation**: Incorrectly classifying captured outer variables as lambda
  parameters (or vice versa) would cause false positives or false negatives. Mitigation: cross-reference `invokedynamic`
  bootstrap args and call-site descriptor.
- [ ] **Kotlin compiler version variance**: Different Kotlin compiler versions may use different synthetic naming
  conventions for lambda classes. Mitigation: rely on interface implementation (`FunctionN`) rather than class name
  patterns.
- [ ] **Desugared lambdas (Android / D8)**: Android toolchains may desugar lambdas differently. Mitigation: document as
  out-of-scope for v1; revisit if user demand arises.
- [ ] **`_` naming instability**: Kotlin's `$noName_<n>` convention and Java 22+'s `_` are compiler implementation
  details that could change. Mitigation: treat `_` and `$noName_` prefixes as intentionally unused in both languages;
  add regression tests with pinned fixtures for both javac 22+ and kotlinc.
- [ ] **Inline lambda debug info dependency**: Detection of inline lambdas requires `LocalVariableTable` to be present.
  If bytecode is compiled with `-g:none` or debug info is stripped, inline lambda analysis produces no findings (safe
  fallback, but reduces coverage). Mitigation: document requirement; most Kotlin builds retain debug info by default.
- [ ] **`$i$a$` marker naming stability**: The `$i$a$` / `$i$f$` convention is a Kotlin compiler implementation detail.
  Mitigation: add pinned fixture tests for specific Kotlin compiler versions; detect markers by prefix pattern.
- [ ] **Multi-catch or complex control flow**: A parameter loaded only in dead code is technically "used" at bytecode
  level. Accept this as a known limitation (no dead-code elimination in this rule).
